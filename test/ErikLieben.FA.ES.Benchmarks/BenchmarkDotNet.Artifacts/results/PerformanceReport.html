7. Recommendations

Critical
Use snapshots for aggregates with >1,000 events
Snapshots bypass event replay entirely, reducing load time from 100+ μs to a single deserialization.
High
Keep event payloads small (<1 KB)
Large payloads (10 KB) take 130x longer to deserialize. Store references to large data instead of embedding it.
Medium
Consider lazy deserialization
Only deserialize events when the When() handler actually needs the data. Skip irrelevant events.
Low
Framework optimizations
Registry lookups (1 ns) and event dispatch (1.2 ns) are already optimal. No further optimization needed here.

8. Conclusions

The framework is extremely well-optimized.
  • Registry lookups use FrozenDictionary (1 ns, 0 allocations)
  • Event folding has zero allocations in the dispatch loop
  • Source-generated JSON serializers eliminate reflection overhead
  • Schema upcasting scales linearly with minimal overhead

The primary performance consideration for users of this framework is event payload size and aggregate stream length. Both are addressed by the built-in snapshot support.

For real-world applications, network I/O to storage providers (Azure Blob, CosmosDB) will dominate total latency, making these microsecond-level optimizations largely academic but still valuable for high-throughput scenarios.
